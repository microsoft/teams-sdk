Teams SDK for .NET 2.1 preview is here. Building on the [v2.0 foundation](https://microsoft.github.io/teams-sdk/csharp/getting-started/quickstart), this release brings deeper ASP.NET Core integration, a layered architecture, and first-class support for agentic identities. If you're building bots for Microsoft Teams in .NET, this is the SDK designed to feel native to how you already write ASP.NET Core applications.

## Native ASP.NET Core Integration

The 2.1 preview is built around the patterns .NET developers already know. There are no custom logging abstractions, no proprietary configuration systems, and no special service containers. Everything plugs into standard ASP.NET Core:

- **Dependency injection** — All services register via `IServiceCollection` using `AddTeamsBotApplication()`
- **Logging** — Standard `ILogger` throughout, configured with `builder.Logging`
- **Configuration** — Standard `IConfiguration` via `appsettings.json`
- **Authentication** — MSAL-based via Microsoft Identity Web, with built-in support for multiple credential types

Here's a minimal bot that handles messages in about 15 lines:

```csharp
using Microsoft.Teams.Apps;

WebApplicationBuilder builder = WebApplication.CreateSlimBuilder(args);
builder.Services.AddTeamsBotApplication();
WebApplication app = builder.Build();

TeamsBotApplication teams = app.UseTeamsBotApplication();

teams.OnMessage(async (context, cancellationToken) =>
{
    await context.SendActivityAsync($"You said: {context.Activity.Text}", cancellationToken);
});

app.Run();
```

`AddTeamsBotApplication()` registers everything the SDK needs into DI. `UseTeamsBotApplication()` wires it into the ASP.NET Core pipeline. From there, you register handlers for the activity types you care about.

### MSAL Authentication

The SDK uses [MSAL (Microsoft Authentication Library)](https://learn.microsoft.com/entra/msal/) for all authentication, replacing the legacy Bot Framework authentication stack. This brings several benefits: automatic token caching, support for modern credential types, and compatibility with the broader Microsoft Identity platform.

Authentication is configured through the standard `AzureAd` section in `appsettings.json`. The SDK auto-detects the credential type based on which fields are present:

## Typed Activity Routing

Instead of a single `OnTurn` callback that switches on activity type, the SDK provides typed handlers that give you the right activity shape and a rich context object:

```csharp
// Message handling with regex routing
teams.OnMessage("(?i)^help$", async (context, cancellationToken) =>
{
    await context.SendActivityAsync("Here are the available commands...", cancellationToken);
});

// Adaptive Card action submissions
teams.OnAdaptiveCardAction(async (context, cancellationToken) =>
{
    string? feedback = context.Activity.Value?.Action?.Data?["feedback"]?.ToString();
    await context.SendActivityAsync($"Got your feedback: {feedback}", cancellationToken);
    return AdaptiveCardResponse.CreateMessageResponse("Feedback received!");
});

// Task module dialogs
teams.OnTaskFetch(async (context, cancellationToken) =>
{
    return TaskModuleResponse.CreateBuilder()
        .WithType(TaskModuleResponseType.Continue)
        .WithTitle("Task Module Demo")
        .WithCard(myAdaptiveCard)
        .Build();
});
```

Every handler receives a `Context<TActivity>` with convenience methods that eliminate boilerplate:

```csharp
await context.SendActivityAsync("text", cancellationToken);        // Send a message
await context.SendActivityAsync(activity, cancellationToken);     // Send a full activity
await context.SendTypingActivityAsync(cancellationToken);         // Typing indicator
context.Log.Info("Processing message");               // Structured logging via ILogger
```

The context also provides lazy-initialized API clients scoped to the current activity's service URL, so you can call Teams APIs directly:

```csharp
teams.OnMessage(async (context, cancellationToken) =>
{
    // Update a previously sent activity via the scoped API client
    var updated = TeamsActivity.CreateBuilder()
        .WithType(TeamsActivityType.Message)
        .WithText("This message was updated!")
        .Build();

    await context.Api.Conversations.Activities.UpdateAsync(
        context.Activity.Conversation.Id,
        previousActivityId,
        updated,
        cancellationToken);
});
```

## Three-Layer Architecture

The SDK is organized into three NuGet packages, each with a clear responsibility:

| Package | Purpose |
|---------|---------|
| **Microsoft.Teams.Core** | Foundation: Activity Protocol implementation, authentication, middleware pipeline, HTTP clients |
| **Microsoft.Teams.Apps** | Framework: typed handlers, activity routing, OAuth flows, Teams API clients, streaming |
| **Microsoft.Teams.Apps.BotBuilder** | Compatibility: drop-in adapter for existing Bot Framework v4 bots |

Most developers will work with `Microsoft.Teams.Apps`, which provides the `TeamsBotApplication` and typed handlers. The Core layer implements the [Activity Protocol specification](https://github.com/microsoft/Agents/blob/main/specs/activity/protocol-activity.md), making the SDK standards-based rather than tied to a specific framework.

## Performance and Modern .NET

The SDK uses `System.Text.Json` with source-generated serialization contexts throughout:

```csharp
[JsonSerializable(typeof(CoreActivity))]
[JsonSerializable(typeof(TeamsActivity))]
internal partial class TeamsActivityJsonContext : JsonSerializerContext { }
```

This means zero-reflection serialization, reduced allocations, and full AOT (Ahead-of-Time) compilation compatibility. The activity model uses a loose schema design — core protocol fields are typed properties, while channel-specific or evolving fields live in a `JsonExtensionData` dictionary. This keeps the model extensible without requiring SDK updates for every new Teams feature.

## OAuth and SSO

The SDK provides a redesigned OAuth flow system built around named connections. Each connection is an independent `OAuthFlow` instance with its own sign-in and failure callbacks, which means your bot can authenticate users against multiple providers — for example, Microsoft Graph and a third-party API — without any routing logic on your part.

```csharp
// Register multiple OAuth connections
var graphFlow = teams.AddOAuthFlow("graph-connection");
var salesforceFlow = teams.AddOAuthFlow("salesforce-connection");

// Each connection has independent sign-in and failure callbacks
graphFlow.OnSignInComplete(async (context, tokenResponse, cancellationToken) =>
{
    await context.SendActivityAsync("Signed in to Microsoft Graph!", cancellationToken);
});

salesforceFlow.OnSignInComplete(async (context, tokenResponse, cancellationToken) =>
{
    await context.SendActivityAsync("Signed in to Salesforce!", cancellationToken);
});
```

Within a handler, initiate sign-in by calling `SignInAsync` on the flow. The SDK handles the full SSO sequence automatically: it attempts silent token acquisition first, and if no cached token is available, it sends an OAuthCard that triggers the Teams SSO flow. Duplicate token exchanges from multiple Teams endpoints (mobile, desktop, web) are deduplicated automatically.

```csharp
teams.OnMessage(async (context, cancellationToken) =>
{
    // Try silent token acquisition; falls back to OAuthCard if needed
    string? token = await graphFlow.SignInAsync(context, cancellationToken);

    if (token is not null)
    {
        // Token available — call the API
        await context.SendActivityAsync("You're already signed in.", cancellationToken);
    }
    // else: OAuthCard was sent, result arrives via OnSignInComplete
});
```

You can also check sign-in status, get tokens silently, or sign the user out:

```csharp
bool isSignedIn = await graphFlow.IsSignedInAsync(context, cancellationToken);
string? token = await graphFlow.GetTokenAsync(context, cancellationToken);
await graphFlow.SignOutAsync(context, cancellationToken);
```

## What You Can Build Today

The preview supports the following scenarios:

- **Messages** — handling, regex routing, updates, deletes, reactions
- **Adaptive Cards** — action submissions with typed value access
- **Task modules** — fetch and submit dialogs
- **Message extensions** — search queries
- **OAuth and SSO** — per-flow sign-in with token exchange support
- **Conversation events** — members added/removed, install/uninstall
- **Streaming** — progressive message updates with rate limiting
- **Proactive messaging** — send messages and replies outside of a turn
- **Agentic identity** — user-delegated tokens for bots acting on behalf of users
- **Targeted messages** — messages visible only to a specific user

## Migration from Bot Framework v4

If you have existing Bot Framework v4 bots, the `Microsoft.Teams.Apps.BotBuilder` package provides a drop-in compatibility layer. Your current `IBot` implementation works unchanged on the new infrastructure — then you can migrate to native handlers at your own pace.

For the full migration walkthrough, see [Bring your Bot Framework .NET projects to Teams SDK 2.1 preview](/blog/bring-botframework-to-teams-sdk-dotnet-2-1).
